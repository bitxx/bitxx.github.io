4. docker中的dtm访问主机的go-zero报错（即dtm官方提供的gozero案例无法跑通）

官网案例
根据官方案例，dtm和gozero都主机部署运行，案例可以跑通。
但是，开发期间，我为了方便，一般除了主项目在本地主机运行，一般像dtm、etcd等，都习惯使用docker-compose来编排。但这种时候你会发现，官方的这个案例无论如何跑不通，一直提示网络问题，不可访问等等。报错内容：xxx Error while dialing dial tcp 127.0.0.1:50081: connect: connection refused

本身也能理解原因，gozero本地将127.0.0.1:50081服务注册到了etdc，但dtm在容器中，拿到gozero的这个注册地址后，在容器中访问127.0.0.1:50081肯定会失败。这是整体原因，但代码层面，到底卡在哪里了？其实这里涉及到grpc的内容，通过跟踪dtm源码发现：

grpc会到etcd中拿到注册地址，并封装到conn的conns中，而conns对外不开放，我们是无法改变里面的地址的。
因此，可以这么说：开发期间，dtm和go-zero最好都在本地环境中部署和操作

如果开发期间，实在想将dtm放在容器，这里有个方式（亲测，可行）：
可以给gozero的每个服务的配置ListenOn设置为：host.docker.internal:端口号，同时在主机的hosts文件中设置：host.docker.internal 127.0.0.1，这样注册到etcd中的地址就是host.docker.internal:端口号,容器就可以访问外部服务了。
